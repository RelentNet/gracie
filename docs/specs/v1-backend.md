## WHAT WE ARE BUILDING

**GA App** is a private internal web platform for Grace & Associates — a federal healthcare IT consulting firm with approximately 30 active clients and an 8-person team. The platform automates client relationship management: meeting capture, AI document generation, task tracking, and team alignment.

**The core loop:**
A client meeting happens → Recall.ai bot records it → transcript delivered in ~10 seconds → AI pipeline generates 6 documents automatically → tasks extracted and written to the task board → team receives a daily 6 AM briefing with everything they need for the day.

**The business problem it solves:**
The GA team currently writes meeting notes manually, drafts follow-up emails by hand, tracks tasks in spreadsheets, and aligns via email threads. At 30 clients with multiple monthly meetings each, this is an enormous hidden time cost. GA App eliminates all of it.

## INFRASTRUCTURE ARCHITECTURE

### Coolify Server Layout
```
Coolify VPS
├── Container: Next.js frontend + API routes (or separate if backend is Fastify)
├── Container: n8n (custom automations, self-hosted)
└── Cloudflare Tunnel (secure cloud-to-server bridge, no open ports)

External Services (not on VPS):
├── Supabase (managed Postgres + pgvector)
├── Cloudflare R2 (file storage)
├── Logto (auth — cloud or self-hosted TBD)
├── Recall.ai (meeting bot API)
├── Resend (SMTP)
└── Microsoft Graph API (calendar)
```

### Three-Layer Data Architecture
```
Layer 1 — Supabase pgvector    ← what the AI queries (semantic search)
Layer 2 — Cloudflare R2        ← where raw files live (docs, transcripts, uploads)
Layer 3 — In-app file browser  ← what humans see (Next.js frontend, presigned URLs)
```

The AI never reads directly from R2. Every document is chunked, embedded, and stored in pgvector on ingest. R2 is for human file access only, via presigned URLs generated by the backend. R2 credentials are never exposed to the frontend.

### n8n Boundary Rule
The backend owns the core pipeline. n8n owns configurable/custom automations only.

```
Backend (core pipeline)              n8n (custom automations)
────────────────────────────         ──────────────────────────────
Meeting bot dispatch                 Weekly client reports
Document generation (6 types)        Monthly fee summaries
6 AM daily sync                      Ad-hoc digest emails
Calendar scanning + dedup            Slack/Teams alerts (future)
Pre-meeting brief generation         Knowledge Base upload notifications
Task extraction + writing            Any new automation Allie requests
Vector embedding pipeline            Cross-client data digests
```

n8n calls the backend API and Claude API. It never touches Supabase or R2 directly.

## DATABASE SCHEMA (design this in full during planning)

The following tables are required. Define columns, types, indexes, and relationships for each.

### Core Tables

**users**
Synced from Logto on first login. Stores role, calendar connection status, Microsoft token (encrypted), last active.

**clients**
Master client list. Canonical name, aliases (array), cadence (weekly/biweekly/monthly/qbr), fee_tier (low/mid/high — admin only), description field, GA company description (FK to settings or denormalized), relationship health score, last meeting date, Google Drive folder URL (legacy, may remove), created_at.

**meetings**
Every detected or manually created meeting. client_id (FK), date_time (Eastern), duration_minutes, meeting_type (enum), meeting_lead_user_id (FK), attendee_user_ids (array), calendar_event_id, video_link, bot_dispatched (bool), bot_job_id (Recall.ai), transcript_received (bool), pipeline_status (enum: scheduled/in_progress/awaiting_transcript/processing/complete/needs_attention/cancelled), pipeline_started_at, pipeline_completed_at, source (calendar/manual).

**documents**
All generated and uploaded documents. meeting_id (FK, nullable), client_id (FK), document_type (enum: post_meeting_analysis/internal_memo/client_summary/task_checklist/internal_email_draft/client_email_draft/pre_meeting_brief/daily_sync/upload/other), r2_key (the R2 object path), file_name, file_size, source_badge (meeting/upload/auto), requires_review (bool), status (enum: ready/needs_review/delivered/archived), uploaded_by_user_id (FK), created_at.

**tasks**
All extracted and manually created tasks. client_id (FK), source_meeting_id (FK, nullable), source_document_id (FK, nullable), description, owner_user_id (FK), due_date, status (enum: open/in_progress/complete), priority_flag (bool), notes (text), archived (bool), days_overdue (computed or stored), created_at, updated_at.

**task_notes**
Append-only notes on tasks. task_id (FK), author_user_id (FK), content, created_at.

**folders**
R2 folder metadata and permissions. client_id (FK, nullable — null = global like knowledge-base), path (R2 prefix), display_name, visibility (enum: all/restricted), allowed_roles (array), created_by_user_id (FK), created_at. This is the permission layer — R2 stores files, Supabase stores who can see what.

**client_notes**
Notes on client profiles (Notes tab). client_id (FK), author_user_id (FK), content, created_at, updated_at.

**client_tabs**
Custom tab configuration per client (or global template). client_id (FK, nullable — null = global template), tab_order (array of tab definitions), created_by_user_id, updated_at.

**daily_syncs**
One record per daily sync generated. date, content (JSON or markdown), generated_at, delivered_at, meeting_ids_included (array).

**pre_meeting_briefs**
One record per brief. meeting_id (FK), content, generated_at, delivered_at, delivered_to_user_ids (array), r2_key.

**pipeline_runs**
Log of every pipeline execution. meeting_id (FK, nullable), source (enum: recall/manual_upload), started_at, completed_at, duration_seconds, documents_generated (int), status (enum: success/failed/partial), error_message.

**knowledge_base_documents**
Global landscape docs. title, description, topic_tags (array), r2_key, file_name, file_size, uploaded_by_user_id (FK), expiration_date (nullable), ai_active (bool), created_at.

**embeddings**
Vector embeddings for all content. source_type (enum: meeting_document/upload/knowledge_base/transcript), source_id (UUID), chunk_index (int), content (text), embedding (vector — pgvector), created_at. Index on embedding column for similarity search.

**settings**
Key-value store for global settings. key (unique string), value (jsonb), updated_by_user_id, updated_at.
Keys include: ga_company_description, business_hours_start, business_hours_end, ai_model, excluded_calendar_keywords, pre_meeting_brief_default_lead_days.

**client_aliases**
For calendar matching. client_id (FK), alias (string). Index on alias for fast lookup.

**meeting_type_rules**
Keyword → meeting type mappings. keyword (string), meeting_type (enum), created_by_user_id.

**master_record_entries**
Chronological MASTER_RECORD per client. client_id (FK), meeting_id (FK), summary (text), created_at.

## AI PIPELINE — FULL SPECIFICATION

### Prompt Construction Order (always in this exact sequence)
```
1. GA company description (from settings)
2. Client description (from clients table)
3. Document type or meeting type being generated
4. Optional consultant context (from upload form or meeting notes)
5. Historical context (top 5 recent meeting summaries from pgvector + any meetings with has_open_items = true)
6. Source content (transcript text or uploaded file content)
```

### Automatic Meeting Pipeline (triggered by Recall.ai webhook)
```
Recall.ai webhook fires (meeting ended, transcript ready)
  ↓
Verify meeting record exists and bot_job_id matches
  ↓
Store raw transcript in R2 under clients/[client-slug]/transcripts/[YYYY-MM-DD].txt
  ↓
Extract text → chunk → embed → store in pgvector (embeddings table)
  ↓
Query pgvector for 5 most recent meeting summaries for this client
  ↓
Query tasks table for any open items from prior meetings (has_open_items context)
  ↓
Generate all 6 documents via Claude API (sequential or parallel — TBD based on rate limits):
    1. Post-Meeting Analysis
    2. Internal Memo
    3. Client-Facing Summary    [requires_review = true]
    4. Task Checklist
    5. Internal Email Draft
    6. Client Email Draft       [requires_review = true]
  ↓
Store each document in R2 under clients/[client-slug]/generated/[YYYY-MM-DD]/
  ↓
Create document records in Supabase documents table
  ↓
Parse Task Checklist → extract structured tasks → write to tasks table
  ↓
Set has_open_items = true on meeting record if tasks were found
  ↓
Update MASTER_RECORD (master_record_entries table) with new summary entry
  ↓
Update pipeline_runs log (status, duration, doc count)
  ↓
Update meeting record: pipeline_status = complete
  ↓
Send in-app notification to attendees: "Documents ready for [Client] — [Date]"
```

### Manual Upload Pipeline (triggered by upload form submission)
```
File received by backend
  ↓
Store raw file in R2 under clients/[client-slug]/uploads/[YYYY-MM-DD]/
  ↓
Create document record (source_badge = upload)
  ↓
Extract text (based on file type — docx/pdf/txt/csv handled differently)
  ↓
Chunk + embed + store in pgvector
  ↓
Determine appropriate output types based on document_type label + consultant context
  ↓
Generate appropriate document subset via Claude API
  ↓
Extract tasks if applicable → write to tasks table
  ↓
Confirm upload and processing to user
```

### Document Generation Rules
- Client-Facing Summary and Client Email Draft: always requires_review = true, never auto-sent
- Internal Email Draft: stored in documents folder, not auto-sent — team retrieves manually
- All generated content: AI must wrap uncertain/hallucinated content in [VERIFY: ...] tags — this is a prompt-enforced rule
- Task extraction: always structured JSON output regardless of input type
- Knowledge Base documents: available as additional context in all Intelligence tab queries

## CALENDAR INTEGRATION — FULL SPECIFICATION

### Scan Loop (cron job, runs every 30 min during business hours)
```
For each connected team member:
  ↓
Call Microsoft Graph API: GET /me/calendarView (next 7 days)
  ↓
For each event:
  Check against client alias list (fuzzy match on title + attendees + description)
  If match found and not already in meetings table → create meeting record
  If match found and already exists → update attendees list, deduplicate
  If ambiguous (multiple client matches) → flag for Admin manual assignment
  If no match → skip (personal/internal meeting)
  ↓
Deduplicate: if multiple team members have same event (same Teams/Zoom URL or same time+attendees):
  → One meeting record, all GA attendees listed
  → bot_dispatched = false (ready for bot dispatch)
```

### Bot Dispatch Logic
```
For each meeting record where:
  - bot_dispatched = false
  - video_link is not null
  - meeting start time is within 5 minutes
  ↓
POST to Recall.ai API with video_link
  ↓
Store bot_job_id on meeting record
  ↓
Set bot_dispatched = true
  ↓
(This prevents any duplicate bots regardless of how many team members are in the meeting)
```

### Post-Meeting Monitoring
```
When meeting end time passes:
  ↓
Monitor for Recall.ai webhook for 90 minutes
  ↓
If webhook received → pipeline starts (see AI Pipeline above)
  ↓
If 90 minutes pass with no webhook:
  Update meeting: pipeline_status = needs_attention
  Send alert to meeting lead via Resend: "No transcript detected for [Client] — [Date]"
  Meeting lead can: dismiss (cancelled/no recording) or upload manually
```

### Presigned URL Pattern
Frontend never calls R2 directly. All file access goes through the backend:
```
GET /api/files/url?key=[r2-key]&action=[get|put]
  → Backend verifies user role has access to this folder path (check Supabase folders table)
  → Generate presigned URL (15-minute expiry)
  → Return URL to frontend
  → Frontend downloads/uploads directly to R2 using the presigned URL
```

### File Move Pattern (copy + delete, invisible to user)
```
POST /api/files/move
  { sourceKey, destinationKey }
  → Backend: S3 CopyObject(source → destination)
  → Backend: S3 DeleteObject(source)
  → Update document record r2_key in Supabase
  → Return success
```

## AUTHENTICATION — LOGTO INTEGRATION

### Flow
```
User visits app → redirected to Logto login screen
  ↓
User clicks "Sign in with Microsoft"
  ↓
Microsoft OAuth 2.0 flow (Logto handles this)
  ↓
Logto returns JWT with user identity + role claims
  ↓
Backend middleware validates JWT on every request
  ↓
Role extracted from JWT claims → permission checks applied
  ↓
First login: backend creates user record in Supabase, initiates calendar connection
```

### Session Management
- Logto handles token refresh silently
- If Microsoft token expires → redirect to Logto login + Admin alert via Resend
- Role changes take effect on next request (no re-login required if handled in JWT claims)

### Backend Middleware Pattern (apply to every API route)
```javascript
// Pseudocode
const { userId, role } = verifyLogtoJWT(req.headers.authorization)
if (!userId) return 401
if (routeRequiresAdmin && role !== 'admin') return 403
req.user = { userId, role }
next()
```

## OPEN TECHNICAL DECISIONS (resolve during planning session)

1. **Backend structure**: Single Next.js app with API routes, or Next.js frontend + separate Fastify/Express backend service? Considerations: long-running pipeline jobs (document generation can take 60+ seconds) may not work well in serverless API routes. A separate backend service with proper job queuing may be necessary.

2. **Job queue**: The pipeline involves long-running async work (embedding, 6 Claude API calls, file writes). What's the job queue strategy? Options: BullMQ with Redis on Coolify, Supabase Edge Functions for lighter tasks, or a simple in-process queue for MVP.

3. **Backend language**: Node.js (consistent with Next.js) vs Python (stronger text extraction ecosystem for .docx/.pdf/.mp3 transcription handling). Recall.ai delivers the transcript text — but processing uploaded audio files (.mp3/.mp4) may need Whisper or similar. Decide now or handle as a Phase 2 concern.

4. **Logto hosting**: Logto cloud (managed) vs self-hosted on Coolify. Self-hosted gives full control and keeps all identity data on your infrastructure. Cloud is simpler to start. Decision needed before auth implementation.

5. **Real-time updates**: Pipeline status updates on the frontend — polling every 30 seconds vs WebSockets vs Supabase Realtime. For MVP, polling is fine. Plan for which approach scales better.

6. **Document generation parallelism**: Generate all 6 documents in parallel (faster, but 6 concurrent Claude API calls) or sequential (slower, easier to debug, better rate limit handling). Claude Sonnet rate limits may constrain parallelism.

7. **Text extraction for uploads**: .docx → mammoth.js, .pdf → pdfjs or pdf-parse, .csv → papaparse, .mp3/.mp4 → Whisper API or leave for Phase 2? Define what's in scope for launch.

8. **Embedding model**: Use Claude's embeddings or OpenAI text-embedding-3-small for pgvector? OpenAI embeddings are currently more established for production RAG. Decision affects embedding pipeline design.

9. **VPS sizing**: What Hetzner or DigitalOcean plan for the Coolify server? n8n + Next.js + backend all need to fit. Starting recommendation: Hetzner CX32 (4 vCPU, 8GB RAM) or CX42 (8 vCPU, 16GB RAM) if budget allows.

10. **n8n database**: n8n needs its own Postgres instance for workflow storage. Share Supabase or run a separate Postgres container on Coolify? Sharing Supabase is simpler but mixing concerns. Separate container is cleaner.

Recall.ai webhook handler, text extraction, pgvector embedding pipeline, Claude API document generation (all 6 types), task extraction, R2 document storage, pipeline status updates, error handling, pipeline history log. Daily sync cron job.
