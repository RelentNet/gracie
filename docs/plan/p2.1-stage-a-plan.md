2. **Health = purely computed, but admin can adjust *individual signals*** (not pin the final score).

   (`date_time ≤ now AND (transcript_received OR pipeline_status='complete')`).

## 2. Relationship-health algorithm (the headline ALGO)

**Output:** integer 0–100 → `relationship_health`, plus `relationship_trend`
(improving/stable/declining). Color bands already defined (`client-display.ts:19`): >90 Strong,
70–90 Steady, <70 At risk.

### Input signals (each normalized to 0–100)
1. **Cadence adherence** *(largest weight)* — `daysSinceLastMeeting / expectedInterval(cadence)`.
   ≤1× → 100; decays linearly 100→0 between 1× and 2×; ≥2× → 0.
   `expectedInterval` days: weekly 7 · biweekly 14 · monthly 30 · qbr 90 · ad_hoc = no expectation (signal dropped).
2. **Meeting recency** — absolute days since `last_meeting_at`: ≤`recencyFullDays` (30) → 100,
   linear to 0 at `recencyZeroDays` (90). (Secondary to cadence; catches long-dormant accounts.)
3. **Open / overdue tasks** — start 100, subtract `overduePenaltyPerTask` (15) per overdue task
   plus `overdueAgePenaltyPerDay` (1) × total overdue-days; floor 0.
4. **Task completion rate** — `complete / (active total)` × 100.

**Missing-signal handling:** a signal with no data (no meetings / ad_hoc cadence / no tasks) is
**dropped and the remaining weights renormalized**, so a client isn't penalized for data it can't have.
Special case: a client with **zero meetings ever** → cadence + recency both score `noMeetingsScore` (default 0)
rather than being dropped (an un-met client *is* unhealthy). `score = round(Σ wᵢ·sᵢ / Σ wᵢ)` over applicable signals.

### Default weights (sum 100) — **operator sets these (Q1)**
| Signal | Default |
|---|---|
| Cadence adherence | **45** |
| Meeting recency | **20** |
| Open/overdue tasks | **20** |
| Completion rate | **15** |

### Trend derivation
New table `client_health_history` (snapshot `score` + `breakdown` jsonb + `computed_at`). Each recompute
inserts a snapshot, then compares the current score to the most recent snapshot ≥ `trendCompareDays` (14) old:
Δ ≥ `trendThreshold` (+5) → improving · Δ ≤ −5 → declining · else stable. No history yet → `stable`.

### Recompute triggers
- **Nightly cron** (new BullMQ repeatable job, mirrors `calendar-scan.every-30m` — `queues.ts:49`,
  registered in `apps/worker/src/index.ts:174`): recomputes every `type='client'`. First run backfills all.
- **Event-triggered** (single-client, deduped via `jobId: health:<clientId>` so bursts collapse):
  - meeting ingested/linked — calendar scan sets `client_id` (`calendar-scan.processor.ts:547/591`);
  - pipeline writes tasks + master-record (`generate.processor.ts:452/463`);
  - manual task create/complete and cadence edit (from the web PATCH routes).

**One job owns both:** the recompute job, per client, also recomputes `last_meeting_at` from `meetings`
(it already queries them), writing `relationship_health`, `relationship_trend`, `last_meeting_at`,
`health_updated_at` in a single `clients` update — so the nightly run doubles as the `last_meeting_at` backfill.

### Configurability
All weights/thresholds/intervals live in one `settings` row `relationship_health_config` (jsonb **object** —
`settings.value` is jsonb; safe alongside the scalar-string settings since no reader globs all keys). Read with a
typed cast + a hardcoded `DEFAULT_HEALTH_CONFIG` fallback (mirrors `parseInternalDomains`' default), so it works
before the row exists and the operator tunes without a deploy.

### Override policy — **per-signal admin adjustment (locked)**
Score is **purely computed** — never a manually pinned final number. But an **admin** may override any
**individual signal's** normalized 0–100 value **with a required reason**, and the final score recomputes
from the adjusted signals. Stored per-client as `clients.health_adjustments` (jsonb), e.g.
`{ "openOverdueTasks": { "value": 85, "reason": "Task X is genuinely long-running", "byUserId": "…", "at": "…" } }`.
In the recompute, `effectiveValueᵢ = adjustmentᵢ?.value ?? computedValueᵢ`. The breakdown shows both the
computed and the effective value with an "adjusted by X" note; an admin can clear an adjustment. Non-admins
see the read-only breakdown only. The same mechanism accepts a future `meetingSentiment` signal.

## 3. Edit-UX plan

- **Per-card Edit toggle** using the existing `CardHeader` `action` slot (already present on the Overview
  Description card): pencil **Edit** → inline form (same card) → **Save / Cancel**.
- **States:** `idle → editing → saving` (inputs disabled + spinner, mirrors the existing loading pattern) →
  success (exit to fresh values) / error (inline `role="alert"`, stays in edit mode, no data lost).
- **Cards that gain Edit:**
  - Overview: **Client details** (name, initials, type, primary_contact, primary_contact_email, cadence,
    contract_number, drive_folder_url) + wire the existing **Description** Edit button.
  - Finance (admin): fee_tier (select), contract_value (number ≥ 0), billing_cadence.
  - Strategy: no new client-field editor (cadence edited on Overview); its manual surface is the notes feed (§4).
- **Permission gating (reuse existing gates — no new permissions):**
  - client-side: `canEdit()` shows editor affordances; Finance edit shown only under `can('finance.view')` (admin).
  - server-side: PATCH requires `isEditor()`; any admin-only field in the body requires `isAdmin()` →
    else **403** (and `updateClient` strips admin fields as defense-in-depth — mirrors `redactClientForRole` on write).
  - viewer: strictly read-only (no affordances rendered, routes reject).
- **Validation:** required `name`; email format; URL format for drive folder; enum values checked against
  `CLIENT_CADENCES`/`CLIENT_TYPES`/`FEE_TIERS`; `contract_value` finite ≥ 0. `updated_at` bumped on every write.

## 4. Add-task & add-note plan

- `POST /api/tasks` (editor) — `{ clientId, description, ownerUserId?, dueDate?, priorityFlag? }`; `status:'open'`, `source_meeting_id:null` (marks it manual vs AI-extracted).

- Data layer: `createTask` + `updateTask` in `lib/data/tasks.ts`. UI: **Add task** button on the Operations

- `PATCH /api/clients/[clientId]/notes/[noteId]` — edit content (author or admin).
- `DELETE …/[noteId]` — delete (author or admin).
- Data layer: `createClientNote` / `updateClientNote` / `deleteClientNote` in `lib/data/client-detail.ts`.
  UI: wire the dead composer (`notes/page.tsx:81`); notes render on the Notes tab **and** the Strategy "Admin Notes" feed.

## 5. Migration plan — `packages/db/migrations/0007_relationship_health.sql`

Additive + idempotent (latest on disk is `0006`). **Hits the SHARED dev+prod Supabase — apply only after
orchestrator confirmation.**

1. `create table if not exists client_health_history (id uuid pk default gen_random_uuid(),
   client_id uuid not null references clients(id) on delete cascade,
   score int not null check (score between 0 and 100), breakdown jsonb,
   computed_at timestamptz not null default now())` + index `(client_id, computed_at desc)`.
2. `alter table clients add column if not exists health_updated_at timestamptz;`
3. `alter table clients add column if not exists health_adjustments jsonb;` — per-signal admin adjustments (Q2).
4. `insert into settings (key, value) values ('relationship_health_config', '<default json>'::jsonb)
   on conflict (key) do nothing;` — `do nothing` (not `do update`) so operator tuning is never clobbered.

No backfill SQL needed for `last_meeting_at` — the first nightly recompute sets it (predicate:
`date_time ≤ now AND (transcript_received OR pipeline_status='complete')`) for every client.
No changes to `tasks`/`client_notes`/`master_record_entries` (existing columns suffice).
