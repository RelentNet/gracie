## 0. Context (cold-start)
The calendar is meetings-first: each meeting is matched to client org(s) by the **email domains** of its
attendees, via the `client_domains` table (domain → org, `unique(lower(domain))` — a domain belongs to exactly
one org). Internal (Grace & Associates) domains and public free-email domains are never client domains.
Two problems to fix in ONE PR:

**A. GA's own tenant domain is treated as external.** `graceandassociatescom.onmicrosoft.com` is Microsoft's
default routing domain for GA's own M365 tenant (i.e. **internal**), but it currently appears as an external
"unknown org" on ~31 meetings. The internal-domain list lives in `settings.internal_email_domains` (currently
just `graceandassociates.com`).

**B. A client can effectively only hold ONE domain via the product.** Real orgs use several domains — e.g.
**IBM** meets from both `ibm.com` and `us.ibm.com`. The *schema* already supports many domains per client
(`client_domains` is 1-org→N-domains) and `createClient({domains})` inserts several at creation time, **but**:
there is **no way to add a domain to an existing client**, and **`linkMeetingOrg` does not register the
meeting's domain on the org**. So after you create an "IBM" org from `ibm.com`, the `us.ibm.com` meetings stay
unassigned forever.

## Scope — Part A: internal-domain fix
1. **Migration `0005`** that updates the `settings.internal_email_domains` row to
   `graceandassociates.com,graceandassociatescom.onmicrosoft.com`. **Match the existing storage format** —
   `0004` wrote the value via `to_jsonb('...'::text)`; check `pg_typeof(settings.value)` on the live DB and how
   `parseInternalDomains` reads it (it splits a comma-separated **string**; the web reader does
   `typeof value === 'string' ? value : null`). Do not break the read path (write the same jsonb/text shape 0004 used).
2. **Reclassification of already-ingested meetings.** After the change, meetings whose only external domains are
   internal or free-email must become `is_internal=true` (linked to the GA `internal` org) and drop out of
   "unassigned"; `graceandassociatescom.onmicrosoft.com` must stop appearing as an "unknown org" anywhere.
   Check whether the scan (`apps/worker/src/processors/calendar-scan.processor.ts`) recomputes `is_internal` +
   links on existing `pipeline_status='scheduled'` calendar meetings on the next sweep. If it does, trigger a
   sweep (the Admin **"Sync now"** button / `POST /api/calendar/sync`) and verify. If it does NOT, add a one-time
   reclassification (or make the scan idempotently recompute), and say so in the PR.
3. Keep the internal list **data-driven in settings** — do not hard-code the onmicrosoft domain into
   `DEFAULT_INTERNAL_DOMAINS` (that constant is only a fallback).

## Scope — Part B: multi-domain clients
1. **Data layer** (`apps/web/lib/data/clients.ts`):
   - `addClientDomain(clientId, domain)`: normalize + lower-case; **reject** free-email (`isFreeEmailDomain`),
     internal (`loadInternalDomains`), and a domain already owned by another org (unique `client_domains`); insert
     the `client_domains` row; then `backfillMeetingsForDomains(db, clientId, [domain])` to retroactively link its
     meetings + set the primary `client_id` where still unassigned.
   - `removeClientDomain(clientId, domain)`: delete the row. Pick the simplest correct behavior for already-linked
     meetings (leave the `meeting_clients` links as history, or recompute) and document the choice in the PR.
   - `listClientDomains(clientId)`.
2. **API**: `GET`/`POST`/`DELETE /api/clients/[clientId]/domains` — editor-gated (`isEditor(await getRequestUser())`),
   error shape mirroring the orgs route. `POST { domain }`, `DELETE { domain }`.
3. **Link-existing enhancement**: when an editor links an existing org to a meeting carrying an unknown external
   domain, **also register that domain on the org** (→ `addClientDomain` + backfill). Surface it as an option in
   the "Link existing" UI (default: add the domain). This is what actually fixes the IBM workflow: linking IBM to
   one `us.ibm.com` meeting registers `us.ibm.com → IBM` and links the rest.
4. **UI — domains manager** on the client/party detail page (under `apps/web/app/(app)/clients/[clientId]/**`):
   list the org's domains, add one (input + surfaced validation errors), remove one. Non-internal orgs only.
   Loading / error / empty states, consistent with the existing client-detail modules.
