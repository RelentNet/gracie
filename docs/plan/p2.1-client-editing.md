## 2. The recommended data model (state this to the operator, let them adjust)

Every field/area falls into one of four input methods — **this classification IS the core of Stage A**:

- **M — Manual:** a person types it in an edit form (facts, contract terms, contacts, cadence, notes, manually-added tasks).
- **AI — AI-written:** produced by the meeting pipeline (documents, master-record entries, transcript-extracted tasks). Stays read-only in these tabs; a manual **override/edit** may be allowed case-by-case.
- **ALGO — Algorithmic:** computed by a defined formula from other data (**relationship_health**, **relationship_trend**). Read-only in the UI, shown with an "auto" badge + "how it's computed" affordance; optional admin override.
- **AUTO — System-derived:** kept in sync automatically (**last_meeting_at** from `meetings`; `days_overdue` computed at query time).

So the answer to the operator's "AI or edit button?" is **both, per field**: manual edit buttons for facts/contract/relationship inputs; AI for meeting-derived artifacts; algorithms for scores. Stage A pins down exactly which is which.

## 3. STAGE A — deliverables (produce these, then STOP for operator sign-off)

Present as a short plan doc (or plan mode). Do not write feature code until the operator approves.

1. **Field-by-field classification table** — every field on every tab, each marked **M / AI / ALGO / AUTO**, and for **M** who may edit (editor vs admin-only). Start from the first-pass in §6 and refine it against the real tab components; flag anything ambiguous for the operator.

2. **Relationship-health algorithm design** — the headline **ALGO**. Propose:
   - **Input signals** with a normalization for each (candidates in §7).
   - **Weights** (sum to 100%) — the operator's call; present a sensible default.
   - **Recompute triggers:** a worker cron (e.g. nightly) **plus** on relevant events (meeting ingested, task created/completed).
   - **Trend derivation:** how `relationship_trend` (improving/stable/declining) is computed — propose a small `client_health_history` table (snapshot score + timestamp) and a rolling compare, or an alternative.
   - **Override policy:** may an admin pin/override the score (with a reason)? Or purely computed? Ask.
   - **Configurability:** put weights/thresholds in `settings` so the operator tunes them without a deploy.

3. **Edit-UX plan** — the per-tab edit affordance (recommend an **"Edit" toggle per card/section** → inline form → Save/Cancel, with validation + error/saving states), field-level permission gating (finance fields admin-only; viewer read-only), and `updated_at` (optional: who edited).

4. **Add-task & add-note plan** — client-scoped **create task** (+ edit/complete/archive) and wiring the dead **notes** composer (add/edit/delete), including where they hook into Operations/Overview/Notes.

5. **Migration plan** (if any) — e.g. `client_health_history`, a `settings` row for health weights, and any new `clients` columns (`health_updated_at`, optional `health_override`/`description` provenance). Additive + idempotent, style of `packages/db/migrations/000X_*.sql`.

**Escalate for a decision** anything where the operator's intent isn't obvious (e.g. can master-record entries be manually added/edited? should `description` get an AI-assist "suggest" button? do leads/prospects need the same finance fields?).

## 4. STAGE B — build scope (after sign-off, to the approved plan)

- **Data layer:** `updateClient(id, patch)` with **field-level permission gating** (finance fields admin-only; mirror `redactClientForRole` on write); `createTask` + `updateTask` (status/archive/edit, client-scoped); client-note create/edit/delete; task-note create.
- **API:** `PATCH /api/clients/[clientId]` (role- + field-gated); `POST/PATCH/DELETE /api/clients/[clientId]/notes`; `POST /api/tasks` + `PATCH /api/tasks/[taskId]`; finance fields flow through the PATCH clients route (admin). Follow the existing route conventions (`runtime='nodejs'`, `getRequestUser()` gate, 401/403/400 shapes).
- **UI:** per-tab **Edit** on Overview (facts + description), Finance (admin inputs), Strategy (whatever is manual); **Add task** (Operations + an Overview shortcut); wire the **Notes** composer; wire the **description** Edit button; relationship-health shows an **auto-computed** badge + "how it's computed" (and override if approved). Loading/error/empty states throughout.
- **Worker:** the **relationship-health recompute** job (cron + event-triggered) implementing the approved algorithm; **`last_meeting_at` sync** (backfill once + keep current on calendar scan / meeting completion).
- **Permissions:** reuse `isEditor()`/`isAdmin()`; viewer is strictly read-only; financial fields admin-only on read AND write.

## 7. Relationship-health algorithm — candidate inputs (for the Stage-A proposal)

Propose a weighted 0–100 score; operator sets the weights. Candidate signals:
- **Cadence adherence** (recommend the largest weight): is the latest meeting within the client's `cadence` window? Overdue → penalty scaled by how overdue.
- **Meeting recency:** days since `last_meeting_at`.
- **Open / overdue tasks:** count + age of overdue tasks for the client (penalty).
- **Task throughput:** completion rate / on-time completion.
- **Engagement:** meeting frequency vs. the client's own baseline.
- **(Phase-2, optional) sentiment:** from the Post-Meeting Analysis docs — flag, don't build now.

**Trend:** compare current score to the score N days ago (from `client_health_history`) → improving / stable / declining by a threshold. Keep every input, weight, and threshold in `settings` so they're tunable without a deploy.
